# 统一申请书生成器交接文档

## 1. 项目定位

本项目是一个基于 `Electron + Vue 3 + Pinia + Vue Router` 的桌面工具，主要用于：

- 批量导入申请人名单
- 编辑统一申请书正文模板
- 在固定区域叠加公章
- 按名单逐页生成 JPG 文件
- 保存导出记录，方便后续追溯

交接目标不是只说明“代码怎么跑”，更重要的是把当前需求范围和实现边界写清楚，避免后续改动时为了改一个点把导入、预览、盖章、导出等其他链路一起改坏。

## 2. 当前需求功能

### 2.1 名单导入

- 支持选择 Excel 文件导入名单
- 默认读取第一个 Sheet
- 优先读取 `template_name` 列，其次兼容 `姓名` / `name` 列
- 每一行生成一份申请书数据，用于页面切换和批量导出

当前实现重点：

- 名单导入不是“仅做展示”，而是整个申请书分页和导出的数据来源
- 导入字段目前核心只依赖 `name`
- 如果后续新增身份证号、店铺名、手机号等字段，必须同步改动：
  - Excel 解析逻辑
  - 前端记录数据结构
  - 模板变量定义
  - 预览替换逻辑
  - 导出渲染逻辑

### 2.2 模板编辑

- 支持在工作台编辑申请书正文
- 支持字体、字号、颜色、行高、缩进、对齐、粗体、斜体、下划线等常见排版能力
- 支持模板内容本地持久化
- 支持按名单切换不同页面预览

当前实现重点：

- 模板编辑内容直接影响预览和导出结果，不能只改编辑态样式而忽略导出态
- 富文本内容经过白名单清洗，不能随意放宽标签和样式，否则可能导致预览与导出不一致
- 文本框位置和默认排版已经做过归一化迁移，不要在没有回归的情况下随意调整默认坐标和尺寸

### 2.3 变量替换

- 当前模板变量核心是 `name`
- 编辑态保留变量标记
- 预览态和导出态替换为实际姓名

当前实现重点：

- 变量链路是“编辑器内容 -> 富文本解析 -> 页面渲染 -> 导出页面截图”
- 这里只要改错一个环节，就会出现预览正常但导出错误，或者编辑正常但预览不替换的问题

### 2.4 盖章配置

- 支持导入印章图片
- 支持在预览区域拖拽、缩放印章位置
- 支持在盖章配置页通过数字输入精确调整位置和尺寸
- 支持盖章配置本地持久化

当前实现重点：

- 盖章区域是全局配置，不是按单页单独配置
- 改动盖章框坐标、拖拽逻辑、缩放逻辑时，必须同时验证：
  - 工作台预览显示是否正常
  - 盖章配置页数值回显是否正常
  - 导出 JPG 中印章位置是否一致

### 2.5 JPG 批量导出

- 支持按当前导入名单逐页导出 JPG
- 支持选择输出目录
- 支持处理非法文件名和重复文件名
- 支持记录导出批次、导出数量、输出目录和时间

当前实现重点：

- 当前导出主链路不是简单前端截图，而是“主进程创建隐藏窗口 -> 进入导出专用页面 -> 截图指定区域 -> 写入 JPG”
- 这条链路涉及主进程、预加载、路由页、store、预览组件联动，任何一端改动都可能影响导出结果
- 后续如果想优化导出速度或改导出方式，必须先确认不影响现有截图区域、缩放比例和页面稳定时机

### 2.6 导出记录

- 支持查看最近导出批次
- 支持查看导出数量、目标目录、导出状态
- 支持打开输出目录

当前实现重点：

- 导出记录是交付和排错的重要辅助信息，不能轻易删除或降级成仅内存态数据

## 3. 核心业务流程

### 3.1 启动流程

1. 应用启动
2. 渲染层执行 `bootstrap`
3. 读取本地模板配置、印章配置、导出历史
4. 恢复工作台编辑和预览状态

### 3.2 正常使用流程

1. 导入 Excel 名单
2. 编辑申请书模板内容
3. 导入并调整印章位置
4. 预览每一页展示效果
5. 选择输出目录
6. 批量导出 JPG
7. 在导出记录页查看结果并打开目录

### 3.3 导出流程重点

1. 前端逐条取名单记录
2. 主进程创建隐藏导出窗口
3. 路由进入 `/capture-export`
4. 导出专用页面根据 token 拉取导出数据
5. 页面渲染稳定后截图
6. 主进程写入 JPG 文件
7. 写入导出记录

这部分是项目里最容易“改一处坏一片”的链路，后续如果不是明确在改导出功能，不要顺手改这里的时序、窗口参数、截图区域计算。

## 4. 关键代码位置

- `src/main/main.js`

- `src/renderer/pages/ExportsPage.vue`

## 5. 后续改代码必须注意

### 5.1 不要只改表面页面

很多功能不是单页面闭环，而是多层联动：

- 改工作台编辑样式，可能影响导出页面渲染
- 改盖章拖拽逻辑，可能影响导出盖章位置
- 改 Excel 解析字段，可能导致变量替换失效
- 改 store 数据结构，可能影响本地配置恢复和导出记录

结论：任何“看起来只是改 UI”或“只是改一个字段”的需求，都要先检查是否影响导入、预览、导出和本地持久化。

### 5.2 不要随便改这些高风险区域

以下区域属于高耦合区，改动前先看完整调用链：

- `src/renderer/stores/app-store.js`
- `src/renderer/components/DocumentPreviewCard.vue`
- `src/main/main.js`
- `src/renderer/utils/rich-text.js`
- `src/renderer/pages/CaptureExportPage.vue`

如果必须修改，至少先回答清楚三个问题：

1. 这个改动会不会影响导出页截图结果？
2. 这个改动会不会影响本地已保存配置恢复？
3. 这个改动会不会影响 Excel 导入后的多页数据渲染？

### 5.3 变更原则

- 优先做最小改动，不要为了一个小需求大面积重构
- 修改数据结构前，先确认旧配置兼容策略
- 修改默认坐标、尺寸、缩放、截图区域前，必须回归导出结果
- 新增模板变量前，必须保证导入、预览、导出三端一起生效
- 如果不是明确需求，不要把已有功能“顺手优化”

## 6. 建议回归检查清单

每次改完代码，至少手工检查以下流程：

1. 应用启动后能正常恢复模板和印章配置
2. Excel 能成功导入名单，并生成多页数据
3. 工作台能正常编辑正文，变量显示正常
4. 切换页面时预览内容正确
5. 印章能正常导入、拖拽、缩放、保存
6. JPG 能成功导出到指定目录
7. 导出后的图片文字、排版、印章位置正确
8. 导出记录页能看到新批次，并可打开目录

如果改动涉及以下模块，需要额外回归：

- 导入相关：重点检查 Excel 字段兼容和空数据处理
- 富文本相关：重点检查预览态和导出态是否一致
- 盖章相关：重点检查拖拽后保存值和导出值是否一致
- 导出相关：重点检查多页连续导出、命名去重、截图完整性

## 7. 当前实现边界

- 当前核心变量只有 `name`
- 当前配置是全局配置，不区分多个项目或多个模板空间
- 导出记录目前以结果展示为主，没有失败重试和明细诊断能力
- 打包目标当前主要是 Windows NSIS 安装包

后续如果要扩展需求，请优先评估是不是会触及以下边界：

- 单变量模板升级为多变量模板
- 全局配置升级为按项目隔离配置
- 单一导出链路升级为多格式导出
- 单页面编辑升级为模板版本管理

这几类需求都不是“小修小补”，需要按专题改造，不能零散穿插到普通需求里。
