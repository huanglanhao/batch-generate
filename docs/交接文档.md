# 批量申请书生成器交接文档

## 1. 项目定位

本项目是一个基于 `Electron + Vue 3 + Pinia + Vue Router` 的桌面工具，主要用于：

- 批量导入申请人名单
- 编辑批量申请书正文模板
- 在固定区域叠加公章
- 按名单逐页生成 JPG 文件
- 保存导出记录，方便后续追溯

交接目标不是只说明“代码怎么跑”，更重要的是把当前需求范围和实现边界写清楚，避免后续改动时为了改一个点把导入、预览、盖章、导出等其他链路一起改坏。

## 2. 当前需求功能

### 2.1 名单导入

- 支持选择 Excel 文件导入名单
- 默认读取第一个 Sheet
- 优先读取 `template_name` 列，其次兼容 `姓名` / `name` 列
- 每一行生成一份申请书数据，用于页面切换和批量导出

当前实现重点：

- 名单导入不是“仅做展示”，而是整个申请书分页和导出的数据来源
- 导入字段目前核心只依赖 `name`
- 如果后续新增身份证号、店铺名、手机号等字段，必须同步改动：
  - Excel 解析逻辑
  - 前端记录数据结构
  - 模板变量定义
  - 预览替换逻辑
  - 导出渲染逻辑

### 2.2 模板编辑

- 支持在工作台编辑申请书正文
- 支持字体、字号、颜色、行高、缩进、对齐、粗体、斜体、下划线等常见排版能力
- 支持模板内容本地持久化
- 支持按名单切换不同页面预览

当前实现重点：

- 模板编辑内容直接影响预览和导出结果，不能只改编辑态样式而忽略导出态
- 富文本内容经过白名单清洗，不能随意放宽标签和样式，否则可能导致预览与导出不一致
- 文本框位置和默认排版已经做过归一化迁移，不要在没有回归的情况下随意调整默认坐标和尺寸

### 2.3 变量替换

- 当前模板变量核心是 `name`
- 编辑态保留变量标记
- 预览态和导出态替换为实际姓名

当前实现重点：

- 变量链路是“编辑器内容 -> 富文本解析 -> 页面渲染 -> 导出页面截图”
- 这里只要改错一个环节，就会出现预览正常但导出错误，或者编辑正常但预览不替换的问题

### 2.4 盖章配置

- 支持导入印章图片
- 支持在“印章文件”位置直接删除已导入的印章文件
- 支持在盖章配置页根据输入内容生成电子印章本地预览
- 支持选择多种盖章字体生成预览，字体来源于 `src/renderer/assets/fonts`
- 支持本地预览区未生成时显示原始模版，生成后直接切换为生成预览
- 支持将当前生成预览直接导出为 PNG 文件，默认文件名取盖章内容
- 支持在预览区域拖拽、缩放印章位置
- 支持在盖章配置页通过数字输入精确调整位置和尺寸
- 支持通过勾选框切换“框内随机位置显示”与“框内居中显示”
- 支持盖章配置本地持久化

当前实现重点：

- 盖章区域是全局配置，不是按单页单独配置
- 盖章配置页现在有三套概念，后续改动不能混淆：
  - 正式印章：顶部“印章文件”和右侧文档预览对应的当前正式印章
  - 本地预览：左侧“盖章生成”区域生成出来的临时预览
  - PNG 导出：把当前本地预览单独导出成文件，不自动替换正式印章
- 当前“生成盖章”只影响左侧本地预览，不应修改顶部“印章文件”、右侧正式预览或全局配置
- 当前“导出PNG印章”只负责把本地预览保存成文件，不应自动写入 `stamp.imagePath`、`stamp.previewUrl` 或覆盖当前正式印章
- 当前正式印章只有两种来源：
  - 用户手动导入的图片
  - 旧逻辑或后续明确需求下保存为正式印章的文字印章
- 盖章字体目前来自 `src/renderer/assets/fonts` 目录内全部字体文件，字体选择会影响本地预览和基于文字生成的印章导出结果
- 盖章弧形文字排版参数目前集中在 `src/renderer/utils/stamp-generator.js`：
  - `GENERATED_SEAL_FONT_SIZE`
  - `GENERATED_SEAL_TEXT_RADIUS`
  - `arcSpanDegrees` 计算
  后续如果继续调“离圆圈距离”“字间距”“字号”，优先只改这里，不要同时改模板图、外圆和导出逻辑
- 删除印章文件属于正式交互能力，不是仅清空展示文案：
  - 点击删除后，需要同时清空 `stamp.imagePath` 和 `stamp.previewUrl`
  - 删除后，盖章配置页文件路径文案和工作台/预览中的印章都要立即消失
  - 删除动作完成后，必须同步持久化，保证重启应用后不会恢复已删除的印章文件
  - 当前删除仅移除已选文件引用，不重置盖章框坐标、尺寸和随机定位开关
- 新增了 `stamp.randomizePosition` 布尔配置：
  - 勾选时，印章不会在框内居中，而是在盖章框内部做稳定随机偏移
  - 不勾选时，印章恢复为框内居中显示
  - 该配置会随本地配置一起保存和恢复
- 随机位置不是每次刷新都重新乱跳，而是基于记录信息生成稳定种子，保证同一页的预览和导出结果一致
- 改动盖章框坐标、拖拽逻辑、缩放逻辑时，必须同时验证：
  - 工作台预览显示是否正常
  - 盖章配置页数值回显是否正常
  - 导出 JPG 中印章位置是否一致
- 改动随机定位逻辑时，必须同时验证：
  - 勾选与取消勾选切换后，预览是否立即生效
  - 保存配置后重启应用，勾选状态是否正确恢复
  - 同一条记录的预览位置和导出位置是否一致
  - 不同页导出时位置允许不同，但必须始终落在盖章框内
- 新增了“印章单图导出”链路：
  - 渲染层通过 `window.applicationFormApi.saveGeneratedStampImage`
  - 预加载层在 `src/preload/preload.js` 暴露该方法
  - 主进程在 `src/main/main.js` 中弹出保存对话框并写入 PNG
  - 该链路与批量文档导出链路不同，不要混用 `writeExportImages` 的目录导出语义
- 改动 `src/main/main.js` / `src/preload/preload.js` 后，开发环境通常需要重启 Electron 才能让新 API 生效；仅刷新渲染页面可能会出现“is not a function”这类旧 preload 未更新的问题

### 2.5 图片批量导出

- 支持按当前导入名单逐页导出 JPG / PNG
- 支持选择输出目录
- 支持处理非法文件名和重复文件名
- 支持记录导出批次、导出数量、输出目录和时间
- 支持统一 A4 图片尺寸导出
- 支持保证 Windows 与 macOS 导出的正文、盖章框、印章位置尽量一致
- 支持避免 Windows 导出半张、截断、只输出局部的问题

当前实现重点：

- 当前导出主链路不是简单前端截图，而是“主进程创建隐藏窗口 -> 进入导出专用页面 -> 截图指定区域 -> 写入 JPG”
- 这条链路涉及主进程、预加载、路由页、store、预览组件联动，任何一端改动都可能影响导出结果
- 当前导出实际支持 `JPG / PNG` 两种格式，后续涉及导出格式时，必须同时验证两种格式的清晰度、黑度和版式一致性
- Windows 导出已针对大尺寸 A4 页面做过兼容处理，不能再退回到“依赖单个超大隐藏窗口完整渲染”的实现，否则容易再次出现只导出半张的问题
- Windows 导出已针对文字偏灰问题增加导出后补墨处理，后续如果继续调清晰度或颜色，必须确认不会把红章、虚线框、正文位置一起改坏
- 后续如果想优化导出速度或改导出方式，必须先确认不影响现有截图区域、缩放比例、页面稳定时机和 Windows 导出完整性

### 2.6 导出记录

- 支持查看最近导出批次
- 支持查看导出数量、目标目录、导出状态
- 支持打开输出目录

当前实现重点：

- 导出记录是交付和排错的重要辅助信息，不能轻易删除或降级成仅内存态数据

## 3. 核心业务流程

### 3.1 启动流程

1. 应用启动
2. 渲染层执行 `bootstrap`
3. 读取本地模板配置、印章配置、导出历史
4. 恢复工作台编辑和预览状态

### 3.2 正常使用流程

1. 导入 Excel 名单
2. 编辑申请书模板内容
3. 导入并调整印章位置
4. 预览每一页展示效果
5. 选择输出目录
6. 批量导出 JPG
7. 在导出记录页查看结果并打开目录

### 3.3 导出流程重点

1. 前端逐条取名单记录
2. 主进程创建隐藏导出窗口
3. 路由进入 `/capture-export`
4. 导出专用页面根据 token 拉取导出数据
5. 页面渲染稳定后截图
6. 主进程写入 JPG 文件
7. 写入导出记录

这部分是项目里最容易“改一处坏一片”的链路，后续如果不是明确在改导出功能，不要顺手改这里的时序、窗口参数、截图区域计算。

## 4. 关键代码位置

- `src/main/main.js`
  - 主进程入口
  - 文件选择、Excel 解析、导出窗口、截图、JPG 写入
- `src/main/store.js`
  - 本地配置和导出历史持久化
- `src/preload/preload.js`
  - 主进程与渲染层桥接 API
- `src/renderer/stores/app-store.js`
  - 核心业务状态中心
  - 名单、模板、盖章、导出流程都在这里串联
- `src/renderer/pages/WorkspacePage.vue`
  - 工作台主页面
  - 模板编辑、名单导入、导出入口集中在这里
- `src/renderer/pages/StampPage.vue`
  - 盖章位置和尺寸配置页面
- `src/renderer/utils/stamp-generator.js`
  - 文字生成印章、本地预览、字体注册、弧形排版和 PNG 导出命名相关核心逻辑
- `src/renderer/pages/ExportsPage.vue`
  - 导出记录查看页面
- `src/renderer/pages/CaptureExportPage.vue`
  - 导出专用截图页面
- `src/renderer/components/DocumentPreviewCard.vue`
  - 最核心预览组件
  - 文本编辑、变量渲染、盖章拖拽、截图区域计算都与它有关
- `src/renderer/utils/stamp-layout.js`
  - 盖章图片缩放和“居中 / 随机”定位的统一计算逻辑
  - 预览与导出共用，后续如果要改盖章在框内的摆放策略，应优先从这里入手

## 5. 后续改代码必须注意

### 5.1 不要只改表面页面

很多功能不是单页面闭环，而是多层联动：

- 改工作台编辑样式，可能影响导出页面渲染
- 改盖章拖拽逻辑，可能影响导出盖章位置
- 改 Excel 解析字段，可能导致变量替换失效
- 改 store 数据结构，可能影响本地配置恢复和导出记录

结论：任何“看起来只是改 UI”或“只是改一个字段”的需求，都要先检查是否影响导入、预览、导出和本地持久化。

### 5.2 不要随便改这些高风险区域

以下区域属于高耦合区，改动前先看完整调用链：

- `src/renderer/stores/app-store.js`
- `src/renderer/components/DocumentPreviewCard.vue`
- `src/main/main.js`
- `src/renderer/utils/rich-text.js`
- `src/renderer/pages/CaptureExportPage.vue`

如果必须修改，至少先回答清楚三个问题：

1. 这个改动会不会影响导出页截图结果？
2. 这个改动会不会影响本地已保存配置恢复？
3. 这个改动会不会影响 Excel 导入后的多页数据渲染？

### 5.3 变更原则

- 优先做最小改动，不要为了一个小需求大面积重构
- 修改数据结构前，先确认旧配置兼容策略
- 修改默认坐标、尺寸、缩放、截图区域前，必须回归导出结果
- 新增模板变量前，必须保证导入、预览、导出三端一起生效
- 如果不是明确需求，不要把已有功能“顺手优化”

## 6. 建议回归检查清单

每次改完代码，至少手工检查以下流程：

1. 应用启动后能正常恢复模板和印章配置
2. Excel 能成功导入名单，并生成多页数据
3. 工作台能正常编辑正文，变量显示正常
4. 切换页面时预览内容正确
5. 印章能正常导入、拖拽、缩放、保存
6. 盖章生成页未生成时显示原始模版，生成后只显示当前本地预览
7. 切换盖章字体后，本地预览是否立即刷新，且不同字体确实有差异
8. 导出 PNG 印章时，默认文件名是否取盖章内容，且不会覆盖当前正式印章
9. 顶部“印章文件”和右侧正式预览不能被下面本地生成预览影响
10. JPG / PNG 能成功导出到指定目录
11. 导出后的图片文字、排版、盖章框、印章位置正确
12. Windows 导出不能出现半张、截断、局部导出
13. Windows 导出的文字清晰度和黑度符合预期，且 JPG / PNG 观感一致
14. 导出记录页能看到新批次，并可打开目录

如果改动涉及以下模块，需要额外回归：

- 导入相关：重点检查 Excel 字段兼容和空数据处理
- 富文本相关：重点检查预览态和导出态是否一致
- 盖章相关：重点检查拖拽后保存值和导出值是否一致
- 盖章生成相关：重点检查本地预览与正式印章边界、字体切换是否真实生效、PNG 单图导出命名是否正确
- 盖章删除相关：重点检查删除后预览是否立即消失、保存后重启是否不会恢复已删除印章、坐标和尺寸是否仍然保留
- 盖章随机定位相关：重点检查勾选状态恢复、框内随机是否越界、取消勾选后是否恢复居中
- 导出相关：重点检查多页连续导出、命名去重、截图完整性

## 7. 当前实现边界

- 当前核心变量只有 `name`
- 当前配置是全局配置，不区分多个项目或多个模板空间
- 导出记录目前以结果展示为主，没有失败重试和明细诊断能力
- 打包目标当前主要是 Windows NSIS 安装包

后续如果要扩展需求，请优先评估是不是会触及以下边界：

- 单变量模板升级为多变量模板
- 全局配置升级为按项目隔离配置
- 单一导出链路升级为多格式导出
- 单页面编辑升级为模板版本管理

这几类需求都不是“小修小补”，需要按专题改造，不能零散穿插到普通需求里。

本项目表面上是“文档编辑 + 图片导出”，实际核心是一个串联了导入、模板、盖章、持久化、隐藏窗口截图导出的完整流程工具。后续维护时，必须先理解需求边界和调用链，再做最小范围改动，避免为了修一个点把其他功能一起改坏。
